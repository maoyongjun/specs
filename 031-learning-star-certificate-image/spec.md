# 功能规格：学习之星奖状图片生成

**功能目录**：`031-learning-star-certificate-image`  
**创建日期**：`2026-05-26`  
**状态**：Batik demo 已实现并生成样图，待生产接入  
**输入**：需要根据“学习之星”底图背景填充学员信息并生成图片。后续会接入圈选学员逻辑和发送微信学员逻辑；本阶段先完成图片生成方案。当前 demo 模板已包含固定文案，SVG 动态层只叠加学员名称、年份数字、发证日期和落款签名；课程名称、荣誉称号和评选标准描述仍按需求口径由模板承载。实现主链路采用 Java 生成 SVG，再通过 Apache Batik `PNGTranscoder` 渲染成 PNG。

## 背景

- 当前问题：业务需要批量生成个性化“学习之星”奖状图片，供后续固定话术后发送给微信学员。
- 当前行为：截图中旧“实践之星”页面是手动页面叠字并截图的形态；当前仓库未定位到该前端页面源码。后端已有图片合成与图片消息发送参考链路，但没有本次奖状的专用生成能力。
- 目标行为：提供可自动化调用的图片生成能力，以底图为背景，按固定文案和动态字段输出一张可发送的图片 URL 或图片字节。服务运行在 Linux 容器或 jar 包环境中，不要求服务器单独安装 Chrome、Chromium 或其他浏览器。
- 非目标：本次不实现圈选学员逻辑，不实现微信发送任务编排，不改变既有 SOP 发送策略，不新增未确认的数据库表。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 生成单个学员奖状图（优先级：P1）

运营或自动任务给定一个学员后，系统能生成该学员专属“学习之星”奖状图片。

**独立测试**：使用固定 Clock、固定学员名和本地底图生成图片，检查图片尺寸、文件非空、核心字段已参与绘制。

**验收场景**：

1. **Given** 学员名称为 `花生酱`，发证日期为 `2026-05-26`，**When** 生成奖状，**Then** 图片中动态叠加 `花生酱`、`2026` 和 `2026年05月26日`，`同学：` 等固定文案由模板保留。
2. **Given** 可读取的底图资源，**When** 生成图片，**Then** 输出图片保持底图尺寸，且编码为微信可发送的 PNG 图片格式。

### 用户故事 2 - 学员名称兜底（优先级：P1）

当不同来源的学员姓名字段不完整时，系统按优先级选择最合适的展示名。

**独立测试**：对姓名解析器分别传入真实姓名、`tuhsu_name`、企微昵称和空值，断言展示名。

**验收场景**：

1. **Given** 真实姓名存在，**When** 生成图片，**Then** 使用真实姓名。
2. **Given** 真实姓名为空但 `tuhsu_name` 存在，**When** 生成图片，**Then** 使用 `tuhsu_name`。
3. **Given** 仅企微昵称 `开开心心学琴的小明` 存在，**When** 生成图片，**Then** 展示名为 `开开心心学...`。

### 用户故事 3 - 纯 Java 容器部署（优先级：P1）

服务部署在 Linux jar 包或容器环境后，后端仍能独立完成 SVG 生成和 PNG 渲染，不依赖浏览器安装或交互式桌面。

**独立测试**：在 `java.awt.headless=true` 下执行生成命令，验证 Batik 可转出 PNG，生成结果可被 `ImageIO` 读取。

**验收场景**：

1. **Given** Linux 容器内只有 JDK、应用 jar 和应用依赖，**When** 执行生成任务，**Then** 可成功输出 PNG。
2. **Given** 服务器未安装 Chrome/Chromium，**When** 执行生成任务，**Then** 仍可完成生成流程。

### 用户故事 4 - 签名可配置（优先级：P1）

模板中的落款签名需要可控：若底图已包含签名则保持原样；若底图不含签名，则支持配置化叠加签名文本。

**独立测试**：分别对“模板含签名”和“模板不含签名”两种场景生成图片，断言签名处理结果符合配置。

**验收场景**：

1. **Given** 底图中已包含签名，**When** 生成图片，**Then** 不额外覆盖模板签名。
2. **Given** 底图不包含签名且已配置签名文本，**When** 生成图片，**Then** 在 SVG 指定坐标渲染签名文本。
3. **Given** 签名配置为空，**When** 生成图片，**Then** 按默认策略失败或回退为模板原样，并记录明确日志。

### 用户故事 5 - 后续发送链路可复用（优先级：P2）

后续接入微信发送时，生成结果能直接作为图片消息发送参数。

**独立测试**：使用 Mock OSS 或本地存储返回 URL，断言发送层读取的是生成后的图片 URL，而不是静态模板 URL。

**验收场景**：

1. **Given** 图片生成成功并返回 URL，**When** 后续发送固定话术和图片，**Then** 图片消息 payload 的 `url` 指向生成后的个性化图片。
2. **Given** 图片生成失败，**When** 后续发送流程执行，**Then** 不发送错误或空白图片，并记录可排查日志。

## 历史问题防漏分析 *(强制)*

- 关键参数来源和赋值时机：
  - `templateImage`：来源于最终确认的学习之星底图资源；渲染前从资源文件、OSS 或配置路径读取，并在 SVG 中以内嵌 base64 或稳定 URI 方式引用。当前 demo 使用的模板已包含固定文案，只保留动态字段空位。
  - `studentDisplayName`：来源于真实姓名 / `tuhsu_name` / 企微昵称；进入 SVG 生成前统一解析并截断。
  - `currentYearText`：来源于 `LocalDate.now(ZoneId.of("Asia/Shanghai"))`；渲染前现算为 `YYYY年`。
  - `courseName`：本需求硬编码为 `开开华彩弹琴6日体验课`；当前模板已烘焙该固定文案，动态层不重复绘制。
  - `honorTitle`：本需求硬编码为 `学习之星`；当前模板已烘焙该固定文案，动态层不重复绘制。
  - `standardDesc`：本需求硬编码为 `完课情况`；当前模板已烘焙该固定文案，动态层不重复绘制。
  - `issueDateText`：来源于发证当天日期；渲染前按 `YYYY年MM月DD日` 生成。
  - `signatureText`：当前 demo 默认叠加 `李瑶院长`，签名字体使用 `Slideqiuhong-Regular.ttf`；若最终底图已含签名，可切换为模板原样模式。
  - `outputImageUrl`：来源于 PNG 编码和上传结果；生成成功后才允许传入发送层。
- 下游读取字段清单：
  - SVG 生成读取 `templateImage`、`studentDisplayName`、`currentYearText`、`issueDateText`、`signatureText` 和布局配置；固定文案读取自模板图，不在动态层重复生成。
  - Batik 渲染读取完整 SVG 文档并输出 PNG 字节或临时文件。
  - 后续微信发送读取生成结果中的 `outputImageUrl`，现有 `JuziUtil.sendJuzi` 图片消息会写入 payload `url`。
- 空对象 / 占位对象风险：
  - 生成请求不得用空 DTO 或空 Map 下传；至少要有可追踪的学员标识和可解析的姓名来源。
  - 底图资源为空、SVG 生成失败、Batik 转码失败、字体不可用或输出图片为空时必须失败返回，不能继续发送。
- 调用顺序风险：
  - 必须先解析所有绘制字段，再生成 SVG，再用 Batik 渲染 PNG，再上传或返回 URL，最后才进入后续发送。
  - 不能先创建发送任务再异步补齐图片 URL。
- 旧逻辑保持：
  - 保持现有微信图片发送 payload 结构不变。
  - 保持后续 SOP / 任务发送的延迟、人工静默、wxsend 预览开关等既有行为不变。
  - 不复用旧“实践之星”文案：课程、称号、评选标准必须使用本需求的新口径。
- 需要用户确认的设计选择：
  - 最终底图是否为“无动态文字”的干净模板。
  - 底图最终尺寸、各文本坐标、字体文件、字号和颜色。
  - 落款签名是否已经烘焙在底图中；如未烘焙，默认签名文字是什么，是否允许配置化覆盖。
  - 输出图片是否统一使用 PNG，以及是否需要同步保存素材中心。

## 边界情况

- 真实姓名、`tuhsu_name`、企微昵称都为空时，展示名兜底为 `同学`，同时记录日志。
- 企微昵称超过 5 个 Unicode code point 时，截取前 5 个并追加 `...`；不按字节截断。
- 姓名中包含 emoji 或控制字符时，应过滤不可绘制字符，避免中文字体绘制异常。
- 文案过长或坐标不足时，优先使用固定布局和缩字号策略，不能让文字覆盖印章、落款或边框。
- Batik 对 SVG 文本布局能力不等同于浏览器 HTML/CSS，换行、缩放、对齐和字体 fallback 必须在生成器中明确控制。
- 字体加载失败、SVG 解析失败、PNG 编码失败、上传失败都应返回明确错误，不发送空白图。
- 批量生成时同一底图和字体可缓存，单个学员失败不应影响其他学员，具体批处理策略待圈选逻辑提供后补充。

## 假设

- 图片生成会落在后端服务或函数中，以便后续批量任务自动调用。
- 目标部署形态是 Java jar / Linux 容器，服务器不会安装 Chrome、Chromium 或其他浏览器。
- Apache Batik 是新的推荐生成方案，字体文件允许随服务资源一起打包，或由容器镜像预装。
- 最终会提供一张固定文案准确、动态位置留白的学习之星底图；当前 demo 使用本地模板图作为排版参考和样图底图。
- 发证日期格式按用户需求固定为 `YYYY年MM月DD日`；如果最终设计图要求空格样式，需要追加纠正记录。
- `tuhsu_name` 是用户需求给出的字段名；代码调研暂未定位到同名字段，后续接入圈选数据时需要再核实真实字段。
