# 功能规格：跨 AI 工具数据库 SQL 操作技能

**功能目录**: `010-database-sql-skill`  
**创建日期**: 2026-05-10  
**状态**: Spec Ready  
**输入**: 用户要求在 `C:\workspace\ju-chat\specs` 编写 Spec Kit 文档，设计一个可供 Codex、Trae、Claude Code 等 AI 工具使用的 skill。该 skill 需要支持通过预配置堡垒机地址、账号、密码访问数据库，也支持直接预配置数据库连接信息，并能快速通过 SQL 脚本操作数据库，同时生成使用说明文档。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 通过堡垒机连接内网数据库（优先级：P1）

使用者已经在本地安全配置文件中配置堡垒机地址、端口、账号、密码或密钥，以及目标数据库地址。AI 工具触发该技能后，可以根据 profile 自动建立堡垒机通道，再访问目标数据库。

**独立测试**：使用一个包含堡垒机配置的测试 profile，执行连接检查命令，验证技能会解析堡垒机配置、建立本地转发端口、尝试连接数据库，并在日志中隐藏账号和密码。

**验收场景**：

1. **Given** profile 配置 `mode=bastion`，**When** 用户要求测试连接，**Then** 技能通过堡垒机建立 SSH 隧道并连接目标数据库。
2. **Given** 堡垒机密码来自环境变量，**When** 执行连接检查，**Then** 技能只读取环境变量值，不把密码写入命令输出或日志。
3. **Given** 堡垒机连接失败，**When** 执行 SQL 前置检查，**Then** 技能停止执行 SQL 并返回可诊断错误。

### 用户故事 2 - 通过预配置数据库 profile 执行 SQL 脚本（优先级：P1）

使用者已经配置数据库类型、地址、端口、库名、账号和密码。AI 工具可以按 profile 读取 SQL 文件，先识别脚本风险，再执行查询类 SQL，并输出结构化结果。

**独立测试**：准备一个只包含 `SELECT` 的 SQL 文件，执行 `run --profile <name> --file <sql>`，验证连接、执行、结果格式化和退出码符合预期。

**验收场景**：

1. **Given** profile 配置 `mode=direct`，**When** 用户要求执行查询脚本，**Then** 技能直接连接数据库并执行 SQL。
2. **Given** SQL 文件只包含 `SELECT` 或 `WITH` 查询，**When** 默认策略为只读，**Then** 技能允许执行并返回结果摘要。
3. **Given** SQL 文件包含多条语句，**When** 执行前解析脚本，**Then** 技能列出语句数量和风险分类。

### 用户故事 3 - AI 工具共享同一个技能包（优先级：P1）

Codex、Trae、Claude Code 等工具需要使用同一套技能说明和脚本，而不是为每个工具重复维护完全不同的逻辑。技能包应以通用 `SKILL.md`、可执行脚本、配置模板和工具适配说明组成。

**独立测试**：检查技能包目录，验证存在通用 `SKILL.md`、脚本目录、配置示例、Codex 元数据和跨工具使用说明。

**验收场景**：

1. **Given** Codex 加载技能目录，**When** 用户要求“用数据库 SQL 技能执行脚本”，**Then** Codex 能根据 `SKILL.md` 找到配置、风险检查和执行流程。
2. **Given** Trae 或 Claude Code 不使用 Codex 的 `agents/openai.yaml`，**When** 用户导入技能，**Then** 仍可通过 `SKILL.md` 和脚本命令完成同样工作流。
3. **Given** 某工具没有原生 skill 目录，**When** 用户按说明配置项目规则或上下文，**Then** AI 仍能引用 `SKILL.md` 的工作流和本地脚本。

### 用户故事 4 - 写操作需要显式授权（优先级：P1）

数据库操作可能影响真实数据。技能默认只能执行只读查询。`INSERT`、`UPDATE`、`DELETE`、`DDL`、存储过程调用等写操作必须同时满足 profile 允许、命令显式允许、AI 已向用户展示执行计划并取得明确确认。

**独立测试**：准备一个包含 `UPDATE` 的 SQL 文件，在默认 profile 下执行，验证技能拒绝执行；打开 `allow_write` 且传入确认参数后，验证技能进入写操作流程。

**验收场景**：

1. **Given** SQL 包含 `UPDATE`，**When** profile 未启用 `allow_write`，**Then** 技能拒绝执行。
2. **Given** profile 启用 `allow_write`，**When** 命令没有写操作确认参数，**Then** 技能仍拒绝执行。
3. **Given** 写操作获得明确确认，**When** 数据库支持事务，**Then** 技能优先在事务中执行，并记录影响行数。

### 用户故事 5 - 使用说明能指导安装、配置和运行（优先级：P1）

使用者需要一份说明文档，能够从零开始配置技能、填写堡垒机和数据库 profile、设置环境变量、让不同 AI 工具调用技能，并理解安全边界。

**独立测试**：阅读 `usage.md`，验证其中包含安装位置、配置文件示例、命令示例、AI 提示词示例、安全策略和排障说明。

**验收场景**：

1. **Given** 用户首次使用技能，**When** 阅读 `usage.md`，**Then** 可以知道技能包应放在哪里、配置文件应写在哪里。
2. **Given** 用户要连接堡垒机后的数据库，**When** 参考配置示例，**Then** 可以填写 bastion 和 database 两部分配置。
3. **Given** 用户要让 AI 执行 SQL 文件，**When** 参考提示词示例，**Then** 可以要求 AI 先分析脚本再执行。

## 边界情况

- 配置文件不存在、profile 名称错误或必要字段缺失。
- 堡垒机不可达、端口被占用、账号密码错误、密钥权限不正确。
- 数据库网络不可达、账号无权限、库名不存在或驱动缺失。
- SQL 文件为空、编码异常、包含多语句、包含事务语句或包含客户端专用命令。
- SQL 包含 `INSERT`、`UPDATE`、`DELETE`、`TRUNCATE`、`DROP`、`ALTER`、`CREATE`、`CALL`、`EXEC` 等高风险操作。
- profile 指向生产环境时，需要更严格确认和结果脱敏。
- 查询结果过大时，需要默认限制行数，并支持导出到 CSV 或 JSON 文件。
- 日志、异常、AI 回复和审计记录中不得泄露明文密码、token、连接串。
- 不同 AI 工具的技能导入机制可能不同，技能核心必须不依赖单一工具私有格式。

## 需求 *(必填)*

- **FR-001**：系统 MUST 在 `C:\workspace\ju-chat\specs` 下创建本 Spec Kit 目录。
- **FR-002**：目标实现 MUST 提供一个通用技能包目录，建议命名为 `database-sql-skill`。
- **FR-003**：技能包 MUST 包含 `SKILL.md`，并在 frontmatter 中说明它用于数据库连接、堡垒机访问、SQL 脚本分析和 SQL 执行场景。
- **FR-004**：技能包 MUST 包含可执行脚本入口，建议为 `scripts/db_skill.py`，用于 profile 查询、连接测试、SQL 分析和 SQL 执行。
- **FR-005**：技能包 MUST 包含配置示例文件，示例中不得出现真实地址、账号或密码。
- **FR-006**：技能 MUST 支持 `direct` 直连数据库模式和 `bastion` 堡垒机隧道模式。
- **FR-007**：配置 profile MUST 能描述数据库类型、主机、端口、库名、用户名、密码来源、默认只读策略、最大返回行数和是否允许写操作。
- **FR-008**：堡垒机配置 MUST 支持主机、端口、用户名、密码来源、私钥路径和本地转发端口策略。
- **FR-009**：敏感信息 MUST 优先通过环境变量或本机私有配置读取，不得要求用户把真实密码提交到 git 仓库。
- **FR-010**：脚本 MUST 在执行 SQL 前对语句做风险分类，至少区分只读查询、DML、DDL、过程调用和未知语句。
- **FR-011**：默认策略 MUST 拒绝执行写操作。
- **FR-012**：写操作 MUST 同时满足 profile 允许写、命令显式允许写和用户明确确认。
- **FR-013**：查询结果 MUST 支持表格摘要输出，并支持 JSON 或 CSV 导出。
- **FR-014**：日志 MUST 记录 profile、数据库类型、脚本路径、语句分类、开始结束时间、影响行数和退出状态。
- **FR-015**：日志和错误信息 MUST 对密码、token、连接串、私钥路径中的敏感部分做脱敏。
- **FR-016**：技能 MUST 至少定义 MySQL 和 PostgreSQL 的实现路径，并保留 SQL Server、Oracle 等数据库适配扩展点。
- **FR-017**：跨工具说明 MUST 覆盖 Codex、Trae、Claude Code 的使用方式；对没有原生 skill 导入能力的工具，MUST 提供项目规则或上下文引用方案。
- **FR-018**：本规格目录 MUST 包含 `usage.md` 作为使用说明文档。
- **FR-019**：实现完成后 MUST 提供自动化测试覆盖配置解析、敏感信息脱敏、SQL 风险分类、默认拒绝写操作和命令参数校验。
- **FR-020**：实现完成后 MUST 在没有真实数据库凭据的情况下可运行离线测试，不依赖生产数据库。

## 成功标准 *(必填)*

- **SC-001**：本目录包含 `AGENTS.md`、`spec.md`、`tasks.md`、`usage.md`、`checklists/requirements.md`。
- **SC-002**：`usage.md` 包含安装、配置、连接测试、SQL 执行、AI 提示词示例、安全策略和排障说明。
- **SC-003**：技能实现后，`profiles` 或等价命令可列出可用 profile，且不会打印密码。
- **SC-004**：技能实现后，`test --profile <name>` 可验证直连或堡垒机连接链路。
- **SC-005**：技能实现后，只读 SQL 在默认策略下可执行，并返回受行数限制的结果。
- **SC-006**：技能实现后，包含 DML 或 DDL 的 SQL 在默认策略下被拒绝。
- **SC-007**：技能实现后，写操作只有在 profile、命令参数和用户确认都满足时才会执行。
- **SC-008**：技能实现后，Codex、Trae、Claude Code 可复用同一个 `SKILL.md` 和脚本工作流。
- **SC-009**：技能实现后，自动化测试可在无真实堡垒机和无真实数据库的环境下通过。

## 假设

- 使用者会在本机安全位置维护真实堡垒机和数据库凭据。
- AI 工具运行环境可以调用本地 Python 或等价脚本运行时。
- SQL 文件由使用者提供，技能只负责分析、执行和输出结果，不负责生成业务正确性保证。
- 生产数据库操作需要使用者自行确认权限、窗口期、备份和回滚方案。
- 不同 AI 工具的技能安装路径可能随版本变化，使用说明以“通用技能包 + 工具适配”方式描述。
