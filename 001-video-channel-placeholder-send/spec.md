## 用户场景与测试 *(必填)*

### 用户故事 1 - 视频号占位符按原文顺序发送（优先级：P1）

运营或 AI 回复内容中包含文本、图片和视频号占位符时，系统必须按照扣子返回内容中的出现顺序拆分并逐段发送。视频号占位符形如 `##{channels:V18}`，其中 `V` 大小写不敏感；发送后的聊天窗口中，视频编号文字和对应的视频号卡片必须相邻且顺序一致。

**优先级原因**：视频号卡片若与前置说明文字错位，会让用户无法判断哪段文字对应哪个视频，直接影响消息可读性和测试验证。

**独立测试**：构造包含 `视频编号： V18`、`##{channels:v18}`、文本尾段和图片占位的扣子回复，调用 `sendJuzi` 后验证发出的消息片段类型、顺序和视频号编码均正确。

**验收场景**：

1. **Given** 扣子回复为 `视频编号： V18\n##{channels:V18}`，**When** `sendJuzi` 处理该内容，**Then** 系统先发送文本消息 `视频编号： V18`，再发送 V18 对应的视频号消息。
2. **Given** 扣子回复包含 `##{channels:v18}`，**When** 系统解析占位符，**Then** 识别结果与 `##{channels:V18}` 完全一致。
3. **Given** 一条回复中连续包含多个视频号占位符，**When** 系统逐段发送，**Then** 每个占位符前后的文本、图片和视频号卡片保持原始顺序。
4. **Given** 底层函数计算调用可能异步处理，**When** 同一次 `sendJuzi` 发送多个片段，**Then** 调用层必须使用串行提交、同步确认或等价机制，确保前一段完成提交后再提交后一段。

### 用户故事 2 - 通过配置和 OSS 原始 JSON 发送视频号卡片（优先级：P1）

系统解析到视频号编码后，应从 `video-channel-batch-config.json` 找到对应配置，再根据配置中的 OSS URL 获取原始 JSON，组装函数计算 `SEND_MESSAGE` 请求体并发送视频号消息。请求体只包含接口需要的业务字段，token 由函数计算内部自动处理。

**优先级原因**：视频号消息的字段较多且来自原始回调 JSON，必须统一从配置和 OSS 数据还原，避免手工复制字段导致不一致。

**独立测试**：准备一个包含 V18 映射的 `video-channel-batch-config.json` 和可读取的 OSS 原始 JSON，验证系统能组装 messageType 为 14 的视频号消息 payload。

**验收场景**：

1. **Given** `video-channel-batch-config.json` 中存在 `V18` 对应 OSS URL，**When** 解析到 `##{channels:V18}`，**Then** 系统读取 OSS 原始 JSON 并构造视频号消息。
2. **Given** OSS 原始 JSON 中包含 `avatarUrl`、`coverUrl`、`description`、`extras`、`feedType`、`nickname`、`objectId`、`objectNonceId`、`thumbUrl`、`url`，**When** 发送视频号消息，**Then** 这些字段进入 `payload`。
3. **Given** 原始 JSON 的 `description` 为空，**When** 系统构造 payload，**Then** 使用视频号编码或其他可追踪兜底文案填充 `description`，不得发送空 `payload.description`。
4. **Given** 函数计算会自动带上 token，**When** 本功能组装请求体，**Then** 请求体不得从原始 JSON 或配置中透传 token。

---

### 用户故事 3 - ai-reply 与 delay-mq 共享解析和组装逻辑（优先级：P1）

开发者在 `fc/ai-reply` 和 `fc/delay-mq` 两个函数中维护消息发送逻辑时，视频号占位符解析、视频号配置解析、OSS 原始 JSON 读取、Redis 缓存和 payload 组装应尽量由 `fc/common` 中的工具类提供，两个函数只负责接入各自现有的 `sendJuzi` 流程。

**优先级原因**：两个函数承担相似消息发送职责，重复实现会造成行为差异，后续修复也容易漏改。

**独立测试**：使用同一段包含文本、图片、视频号的输入，分别验证 `ai-reply` 和 `delay-mq` 产出的消息片段列表、视频号编码和 payload 字段一致。

**验收场景**：

1. **Given** `ai-reply` 和 `delay-mq` 收到相同扣子回复，**When** 两边调用各自 `sendJuzi`，**Then** 拆分结果和每个片段的发送类型一致。
2. **Given** 视频号 payload 组装规则需要调整，**When** 修改共享工具类，**Then** 两个函数模块无需复制同一段业务规则。
3. **Given** 现有文本和图片发送已通过 `sendJuziTextOrImage` 实现，**When** 增加视频号支持，**Then** 文本和图片发送格式保持兼容。

### 用户故事 4 - OSS JSON 读取具备 Redis 缓存和降级处理（优先级：P2）

同一个视频号编码可能被多次发送。系统应将从 OSS URL 获取到的实际 JSON 缓存到 Redis 30 分钟，缓存命中时不再访问 OSS；缓存不可用或 OSS 读取失败时，应记录可定位日志并避免发送 malformed 视频号消息，同时不阻塞同一回复中的其他有效文本或图片片段。

**优先级原因**：视频号批量发送会频繁读取同一批 OSS JSON，没有缓存会增加外部依赖压力；异常处理不当会导致整段回复发送失败。

**独立测试**：连续两次发送同一编码，验证第一次访问 OSS 并写入 Redis，第二次在 30 分钟 TTL 内命中 Redis；再模拟 OSS 失败，验证其他文本片段仍可发送。

**验收场景**：

1. **Given** Redis 中没有 V18 对应的原始 JSON 缓存，**When** 系统发送 V18，**Then** 读取 OSS URL 并写入 30 分钟 TTL 的 Redis 缓存。
2. **Given** Redis 中存在未过期的 V18 缓存，**When** 系统再次发送 V18，**Then** 直接使用缓存数据，不访问 OSS。
3. **Given** OSS 返回 gzip 或其他常见压缩响应，**When** 系统读取原始 JSON，**Then** 能正确解压并解析 JSON。
4. **Given** 视频号编码不存在、OSS 读取失败或原始 JSON 缺少必填字段，**When** 系统处理该占位符，**Then** 不发送 malformed 视频号消息，记录编码和失败原因，并继续处理其他有效片段。

---

### 用户故事 5 - 同一客户同一员工同一视频号 7 天内限发一次（优先级：P1）

运营或 AI 回复内容中重复命中同一个视频号编码时，系统应避免在短时间内反复向同一个外部联系人发送同一张视频号卡片。限发维度为 `external_user_id + user_id + 视频号编码`，同一维度下 7 天内只允许成功提交一次视频号卡片。

**优先级原因**：重复发送同一视频号会造成用户打扰和消息冗余，但限发不能影响其他文本、图片或不同视频号内容的正常发送。

**独立测试**：构造同一回复中包含两次 `##{channels:v18}` 的内容，验证第一次视频号可发送，第二次视频号被跳过，前后文本仍继续发送；再验证不同外部联系人、不同企微员工或不同视频号编码互不影响。

**验收场景**：

1. **Given** 同一 `external_user_id + user_id + V18` 在 Redis 中没有 7 天限发记录，**When** 系统准备发送 V18，**Then** 使用 Redis `NX + EX` 写入 604800 秒占位并继续提交函数计算。
2. **Given** 同一 `external_user_id + user_id + V18` 已存在未过期限发记录，**When** 同一回复或后续回复再次解析到 V18，**Then** 只跳过该视频号卡片，并继续发送其他有效文本、图片和不同视频号片段。
3. **Given** V18 函数计算同步提交失败，**When** 系统捕获发送异常，**Then** 删除本次 7 天占位，允许后续重试。
4. **Given** Redis 限发检查、占位或释放失败，**When** 系统处理视频号片段，**Then** 记录可定位日志并继续发送视频号，不阻断回复完整性。

---

### 用户故事 6 - 条件文本跟随后面视频号发送决策（优先级：P1）

运营或 AI 回复内容中可以使用 `##{text:<content>}` 表示一段与紧邻后一个视频号绑定的条件文本。系统应在发送前判断该视频号是否会发送：如果视频号会发送，则先发送条件文本，再发送视频号；如果视频号不会发送，则条件文本也不发送。普通文本不受该规则影响。

**优先级原因**：视频号被 7 天限发或配置异常跳过时，用户不应只收到孤立的视频说明文字；但同一回复中非条件普通文本仍应正常触达。

**独立测试**：构造 `正常发送的文字##{text:这个文字会根据后面的视频号是否发送过，决定是否发送}##{channels:V18}`，分别验证 V18 可发送和 7 天限发命中时的发送片段。

**验收场景**：

1. **Given** 回复为 `正常发送的文字##{text:条件文字}##{channels:V18}` 且 V18 未触发 7 天限发，**When** `sendJuzi` 处理该内容，**Then** 系统依次发送普通文本 `正常发送的文字`、条件文本 `条件文字`、V18 视频号。
2. **Given** 同一 `external_user_id + user_id + V18` 已存在 7 天限发记录，**When** `sendJuzi` 处理 `正常发送的文字##{text:条件文字}##{channels:V18}`，**Then** 系统只发送普通文本 `正常发送的文字`，跳过条件文本和 V18 视频号。
3. **Given** `##{text:条件文字}` 后面没有紧邻 `##{channels:<code>}`，**When** 系统解析并发送，**Then** 跳过该条件文本，并继续处理后续其他有效文本、图片或视频号片段。
4. **Given** 条件文本绑定的 V18 因配置缺失、OSS 失败、JSON 解析失败或 payload 构建失败无法发送，**When** 系统处理绑定组合，**Then** 条件文本和 V18 都不发送，并继续发送其他有效片段。
5. **Given** 条件文本已发送但绑定视频号函数计算同步提交失败，**When** 系统捕获异常，**Then** 释放本次 7 天限发占位，但不回滚已发送条件文本，以保持正常情况下文本在视频号前面的顺序。

### 边界情况

- 占位符 `##{channels:V18}` 与普通文本紧邻，没有空格或换行。
- 条件文本 `##{text:...}` 与普通文本、视频号占位符紧邻，没有空格或换行。
- 占位符使用小写 `v`，例如 `##{channels:v18}`。
- 同一回复中包含多个视频号占位符、图片占位符和普通文本。
- 同一回复中包含多个 `##{text:...}##{channels:...}` 条件文本与视频号组合。
- `##{text:...}` 后面不是紧邻视频号，而是普通文本、图片、另一个条件文本或回复结束。
- `##{text:}`、`##{text:abc` 或条件文本内容包含 `}` 等 malformed 条件文本占位符。
- 同一视频号编码在一条回复中重复出现。
- 同一视频号编码在 7 天内对同一外部联系人和同一企微员工重复出现。
- 同一视频号编码对不同外部联系人或不同企微员工出现。
- 占位符格式不完整，例如 `##{channels:}`、`##{channels:abc}` 或缺少 `}`。
- `video-channel-batch-config.json` 中不存在对应编码。
- 配置中存在编码，但 OSS URL 为空、超时、返回非 JSON、返回压缩 JSON 或返回缺少必填字段的 JSON。
- 原始 JSON 的 `description` 为空。
- Redis 不可用、写入失败、读到非法缓存、TTL 设置失败或 7 天限发 NX 占位失败。
- 7 天限发占位成功后，函数计算同步提交视频号失败。
- 函数计算底层异步发送导致消息到达顺序可能变化。
- `empExternalDto.getType()` 不是句子发送类型时，原有分支行为需要保持不变。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统 MUST 在 `sendJuzi` 链路识别视频号占位符，格式为 `##{channels:<code>}`，其中 `<code>` MUST 支持 `V` 或 `v` 加数字，例如 `V18`、`v18`。
- **FR-002**：系统 MUST 将识别到的视频号编码统一规范化为大写形式，例如 `v18` 规范化为 `V18`。
- **FR-003**：系统 MUST 按扣子回复原文顺序拆分文本、图片和视频号片段，并按该顺序发送。
- **FR-004**：同一次 `sendJuzi` 内的多个片段 MUST 串行提交或使用等价顺序保证机制，确保文本说明与对应视频号卡片不发生错位。
- **FR-005**：系统 MUST 保持现有文本和图片发送行为兼容，继续支持已有图片占位符和 Markdown 图片格式。
- **FR-006**：视频号消息 MUST 使用消息类型 `14`，即视频号消息类型。
- **FR-007**：视频号 payload MUST 包含接口所需字段：`avatarUrl`、`coverUrl`、`description`、`extras`、`feedType`、`nickname`、`objectId`、`objectNonceId`、`thumbUrl`、`url`。
- **FR-008**：当原始 JSON 的 `description` 为空时，系统 MUST 设置非空兜底描述，兜底值至少能追踪到视频号编码。
- **FR-009**：系统 MUST 从 `video-channel-batch-config.json` 中根据视频号编码获取 OSS URL，并使用该 URL 读取实际原始 JSON。
- **FR-010**：系统 MUST 对 OSS 获取到的实际原始 JSON 做 Redis 缓存，缓存有效期为 30 分钟。
- **FR-011**：Redis 缓存 key MUST 能稳定区分视频号编码和 OSS URL，避免不同 URL 的同名编码互相污染。
- **FR-012**：系统 MUST 支持读取 gzip 或常见压缩传输的 OSS JSON 响应。
- **FR-013**：函数计算请求体 MUST 继续使用 `functionCode = "SEND_MESSAGE"`，并保留现有 `type`、`taskObj` 等调用约定。
- **FR-014**：函数计算请求体 MUST 不手动携带 token；token 由函数计算内部自动处理。
- **FR-015**：共享的解析、配置读取、OSS JSON 获取、Redis 缓存和视频号 payload 组装逻辑 SHOULD 优先放入 `fc/common`，供 `fc/ai-reply` 与 `fc/delay-mq` 复用。
- **FR-016**：`fc/ai-reply` 与 `fc/delay-mq` MUST 对相同输入产生一致的拆分结果和视频号发送 payload。
- **FR-017**：当视频号编码不存在、OSS 读取失败、JSON 解析失败或必填字段缺失时，系统 MUST 记录包含编码和原因的日志，跳过该 malformed 视频号片段，并继续处理其他有效片段。
- **FR-018**：非句子发送分支或现有特殊分支 MUST 保持原有行为，不因本功能改变文本发送、图片发送或其他消息类型的结果。
- **FR-019**：后续实现 MUST 补充自动化测试，覆盖占位符解析、大小写规范化、顺序发送、视频号 payload 组装、Redis 30 分钟缓存、OSS 异常和两个函数模块的一致性。
- **FR-020**：系统 MUST 限制同一 `external_user_id + user_id + 视频号编码` 在 7 天内只成功提交一次视频号卡片。
- **FR-021**：7 天限发记录的 Redis key MUST 使用 `juzi:video-channel:sent:{externalUserId}:{userId}:{CODE}`，其中 `{CODE}` MUST 统一大写，TTL MUST 为 604800 秒。
- **FR-022**：系统 MUST 在视频号 taskObj 构建成功后、函数计算同步提交前，使用 Redis `NX + EX` 原子写入 7 天限发占位；占位失败时 MUST 跳过该视频号卡片。
- **FR-023**：函数计算同步提交视频号失败时，系统 MUST 删除本次 7 天限发占位，允许后续重试。
- **FR-024**：Redis 限发检查、占位或释放异常时，系统 MUST 记录日志并继续发送视频号，不因 Redis 故障破坏回复完整性。
- **FR-025**：7 天限发命中时，系统 MUST 只跳过重复视频号卡片，继续按原文顺序发送其他有效文本、图片和未命中的视频号片段。
- **FR-026**：自动化测试 MUST 覆盖 7 天限发 key 维度、大小写归一、604800 秒 TTL、首次允许、重复拦截、失败释放、Redis 异常继续发送和双模块一致性。
- **FR-027**：系统 MUST 在 `sendJuzi` 链路识别条件文本占位符，格式为 `##{text:<content>}`，解析后片段类型为条件文本，发送内容为 trim 后的 `<content>`。
- **FR-028**：条件文本 MUST 只绑定后面紧邻的一个视频号片段；若后面不存在紧邻视频号片段，系统 MUST 跳过该条件文本并继续处理后续有效片段。
- **FR-029**：当条件文本绑定的视频号因 7 天限发命中、配置缺失、OSS 读取失败、JSON 解析失败或 payload 构建失败而不会发送时，系统 MUST 同时跳过条件文本和该视频号。
- **FR-030**：当条件文本绑定的视频号通过预检且可发送时，系统 MUST 先发送条件文本，再发送绑定视频号，保持用户看到的说明文字在视频号卡片之前。
- **FR-031**：当条件文本已发送但绑定视频号函数计算同步提交失败时，系统 MUST 释放本次 7 天限发占位，但不承诺回滚已发送条件文本。
- **FR-032**：`##{text:...}` 不得改变普通文本、图片、独立视频号或 `type != 1` 分支的现有发送行为。
- **FR-033**：自动化测试 MUST 覆盖条件文本解析、绑定组合发送、7 天限发命中跳过条件文本、未绑定条件文本跳过、视频号构建失败跳过条件文本和双模块一致性。

### 关键实体

- **扣子回复内容**：AI 返回给 `sendJuzi` 的原始字符串，可能同时包含普通文本、图片占位符、Markdown 图片和视频号占位符。
- **消息片段**：从扣子回复内容中按顺序拆分出的最小发送单元，类型包括文本、图片和视频号。
- **视频号占位符**：形如 `##{channels:V18}` 的标记，用于引用一个视频号配置编码。
- **条件文本占位符**：形如 `##{text:说明文字}` 的控制型文本标记，只在后面紧邻视频号会发送时发送其说明文字。
- **视频号配置项**：`video-channel-batch-config.json` 中按编码维护的配置，至少包含视频号编码和 OSS 原始 JSON URL。
- **OSS 原始 JSON**：从配置 URL 获取的实际视频号消息数据，包含头像、封面、描述、昵称、视频 URL 等 payload 字段。
- **Redis 缓存项**：OSS 原始 JSON 的 30 分钟缓存，减少重复 OSS 请求。
- **视频号 7 天限发记录**：按外部联系人、企微员工和视频号编码生成的 Redis 占位记录，用于避免 7 天内重复发送同一视频号卡片。
- **SEND_MESSAGE 请求体**：传给函数计算的业务请求体，包含 `functionCode`、消息接收人信息、消息类型和对应 payload。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：输入 `视频编号： V18\n##{channels:v18}` 时，系统 100% 先发送文本 `视频编号： V18`，再发送 V18 视频号消息。
- **SC-002**：包含 18 个视频号占位符和 18 段编号文本的批量回复中，所有文本编号与视频号卡片一一相邻且顺序一致。
- **SC-003**：`ai-reply` 和 `delay-mq` 对同一段包含文本、图片和视频号的输入，产生一致的消息片段序列和视频号 payload。
- **SC-004**：同一视频号编码在 30 分钟内重复发送时，第一次读取 OSS，后续请求命中 Redis 缓存，不再重复访问 OSS。
- **SC-005**：原始 JSON 的 `description` 为空时，视频号发送不会再出现 `payload.description is empty`。
- **SC-006**：配置缺失、OSS 失败或 JSON 非法时，系统记录错误并继续发送同一回复中的其他有效文本或图片片段。
- **SC-007**：现有文本和图片发送相关自动化测试保持通过，无新增回归。
- **SC-008**：同一 `external_user_id + user_id + V18` 在 7 天内重复出现时，只发送第一次 V18 视频号卡片，重复命中的视频号卡片被跳过，其他文本和图片继续发送。
- **SC-009**：同一 V18 对不同外部联系人、不同企微员工，或同一用户发送 V19 时，不受 V18 的 7 天限发记录影响。
- **SC-010**：V18 函数计算同步提交失败后，本次限发占位被释放，下一次 V18 发送可重新尝试。
- **SC-011**：输入 `正常发送的文字##{text:条件文字}##{channels:v18}` 且 V18 可发送时，系统 100% 依次发送普通文本、条件文本和 V18 视频号。
- **SC-012**：同一输入在 V18 7 天限发命中时，系统 100% 只发送普通文本，条件文本和 V18 视频号均不发送。
- **SC-013**：`##{text:条件文字}` 后面未紧邻视频号时，条件文本不会作为普通文本泄露给用户，后续其他有效片段继续发送。

## 假设

- `video-channel-batch-config.json` 继续作为视频号编码到 OSS 原始 JSON URL 的配置来源；后续如迁移到配置中心，应保持同等字段语义。
- 视频号编码格式限定为 `V` 加数字，系统内部统一使用大写编码。
- `fc/ai-reply` 和 `fc/delay-mq` 当前都能依赖 `fc/common` 中新增的共享工具类或接口。
- Redis 客户端连接能力由现有函数模块提供；若 common 不直接依赖具体 Redis 实现，可通过适配接口接入。
- 函数计算 `SEND_MESSAGE` 内部负责自动追加 token，本功能不处理 token 获取和刷新。
- 7 天限发维度限定为同一外部联系人、同一企微员工、同一视频号编码；重复命中只跳过视频号卡片，不删除相邻说明文本。
- `##{text:...}` 是控制型条件文本，不是普通文本占位；只有紧邻后一个视频号会发送时才发送。
- 为保持消息顺序，条件文本在绑定视频号前发送；若视频号函数计算提交失败，不回滚已发送条件文本。
