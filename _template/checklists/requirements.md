## 内容质量

- [ ] 明确目标项目、模块、入口和核心实现位置。
- [ ] 明确用户目标、成功标准和非目标。
- [ ] 明确新增、修改和禁止改变的行为。
- [ ] 明确日志、时间、延迟、幂等、fallback、兼容性或异常处理要求。
- [ ] 明确后续实现必须增加测试或静态验证记录。

## 需求完整性

- [ ] 无 `[NEEDS CLARIFICATION]` 或未替换占位内容残留。
- [ ] 需求可测试且无明显歧义。
- [ ] 成功标准可衡量。
- [ ] 验收场景覆盖正常路径、边界路径和不回归路径。
- [ ] 边界情况已识别，并明确跳过、兜底、抛错或记录日志的策略。

## 参数完整性门禁

- [ ] 已列出关键参数来源和赋值时机。
- [ ] 已列出下游读取字段清单。
- [ ] 没有未解释的 `new XxxDto()`、空 JSON、空 Map 或占位参数。
- [ ] 下游读取字段在调用前已赋值，或在当前层现算现用。
- [ ] 不存在未处理的调用后赋值风险。
- [ ] 外部接口、Feign、FC、MQ、Redis 或数据库写入的关键参数已有下游参数断言方案。
- [ ] 若修复会改变调用顺序、接口契约、远程调用或业务语义，已记录并完成用户确认。

## 实施就绪度

- [ ] 实现范围已限定，不扩散到无关模块。
- [ ] 不新增数据库表、不新增对外 API、不修改 MQ/Redis/配置契约，除非规格明确要求。
- [ ] 已确认旧逻辑中必须保持不变的过滤、异常、日志、延迟和 fallback。
- [ ] 每个关键需求至少有一条测试、编译或静态验证任务。
- [ ] 单元测试计划避免真实访问 Redis、OTS、Center、RocketMQ、FC 或外部 HTTP，除非规格明确要求联调。
- [ ] 补充需求或纠正需求时，已同步更新 `spec.md`、`tasks.md` 和 `AGENTS.md`。

## 备注

- 强制门禁未完成前，不进入实现。
- 发现模型未识别的历史代码风险时，先补文档和测试映射，再改代码。
