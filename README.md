# specs

本目录记录需求规格、实现任务、验证记录和后续纠正记录。新需求默认从 `_template` 复制，不直接从历史需求目录复制。

## 新需求创建流程

1. 复制 `specs/_template`。
2. 重命名为下一个编号目录，例如 `026-feature-name`。
3. 先填写 `AGENTS.md` 和 `spec.md`，明确目标、范围、参数来源和旧逻辑约束。
4. 实现前完成 `tasks.md` 的 Phase 1“代码事实确认”和 Phase 2“风险门禁”。
5. 实现前完成 `checklists/requirements.md`，特别是参数完整性门禁。
6. 实现后记录测试命令、测试结果、自检结论和剩余风险。
7. 每次用户补充或纠正需求，都追加 Dxxx 执行记录，并同步更新 `spec.md`、`tasks.md`、`AGENTS.md`。

## 强制门禁

- 关键参数必须追踪来源和赋值时机。
- 下游读取字段必须在调用前有值，或在当前层现算现用。
- 不允许未解释的空 DTO、空 JSON、空 Map 或占位对象继续下传。
- 涉及调用顺序、接口契约、远程调用、MQ、Redis、数据库或业务语义变化时，必须先确认。
- 测试不能只断言最终结果；涉及外部调用时必须做下游参数断言。
