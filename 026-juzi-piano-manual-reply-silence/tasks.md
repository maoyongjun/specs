# 任务清单：钢琴用户级人工回复静默（AI/SOP）

**输入**：来自 `spec.md` 的功能规格  
**前置条件**：`spec.md`、`AGENTS.md`、`checklists/requirements.md`  
**测试**：实现阶段必须补充与关键行为一一对应的测试或静态验证记录。

## Phase 1：代码事实确认

- [ ] T001 复查本次需求与 `AGENTS.md`，确认范围只覆盖钢琴用户级静默。
- [ ] T002 搜索 `juzi-service`、`fc/ai-reply`、`fc/sop-reply` 的真实入口、调用链、发送点和 key 来源。
- [ ] T003 确认 `skuId` 的来源和赋值时机，特别是自发消息早退路径是否会漏写静默锁。
- [ ] T004 确认 `ai-reply` 和 `sop-reply` 的最终发送点分别在哪里，哪些 helper 会真正发消息。
- [ ] T005 确认现有消息级 `removeCache`、`redisKey`、`recentProcess` 和 `delay-mq` 行为必须保持不变。

**检查点**：不得在未完成 T001-T005 前进入实现。

## Phase 2：风险门禁

- [ ] T006 检查是否存在空 DTO、空 JSON、空 Map 或占位参数来承载静默状态。
- [ ] T007 检查是否存在调用后赋值、异步后赋值或早退路径导致静默锁漏写。
- [ ] T008 检查 `sop-reply` 是否误加了额外的 `skuId == 4` 分支，而不是复用共享 key。
- [ ] T009 检查 AI 和 SOP 是否都做到了入口检查加最终发送前检查。
- [ ] T010 为每个关键行为建立测试映射，至少覆盖正常路径、边界路径和不回归路径。

**检查点**：T006-T009 必须有明确结论；发现高风险时先更新 `spec.md` 的“历史问题防漏分析”。

## Phase 3：实现

- [ ] T011 按规格实现静默锁写入和读取。
- [ ] T012 保持未声明的旧行为不变。
- [ ] T013 将所有最终发送点统一接入同一静默拦截。

## Phase 4：测试与验证

- [ ] T014 新增或更新单元测试，覆盖人工回复写锁、AI 拦截、SOP 拦截和 TTL 刷新。
- [ ] T015 单元测试必须断言关键下游参数和拦截结果，不只断言最终返回值。
- [ ] T016 验证非钢琴和 `delay-mq` 不回归。
- [ ] T017 运行目标模块测试或编译命令，并记录结果。
- [ ] T018 搜索确认没有残留旧的单点拦截口径或漏掉的发送点。
