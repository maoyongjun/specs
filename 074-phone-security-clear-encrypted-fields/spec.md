# 功能规格：生产手机号安全字段清空

**功能目录**：`074-phone-security-clear-encrypted-fields`  
**创建日期**：`2026-06-11`  
**状态**：Executed  
**输入**：基于 `069-phone-security-backfill-governance` 最终 DDL，将生产库中已写入的手机号安全字段 `phone_md5/phone_aes/phone_mask` 及等价字段组清空，后续重新回填；候选行按 `*_md5` 非空核对；要求使用 `database-sql-skill`，不要造成线上锁表；当前约 1 万条有值。

## 背景

- 当前问题：线上 `*_mask / *_md5 / *_aes` 安全字段已有值，需要先清空，后续重新按新逻辑回填。
- 当前行为：`069` 已完成 44 张表、45 组手机号安全字段的 DDL 与索引治理，生产库字段和索引已存在。
- 目标行为：仅将目标安全字段置为 `NULL`，保留源手机号字段和表结构不变；候选行只按同一字段组的 `*_md5 IS NOT NULL AND *_md5 <> ''` 判断。
- 非目标：不执行 DDL，不修改应用代码，不删除原手机号列，不导出安全字段值，不本地备份敏感值。

## 用户场景与测试

### 用户故事 1 - 生产安全字段清空（优先级：P1）

运维人员需要在生产库中清空所有手机号安全字段，使后续回填任务可以重新生成 `mask/md5/aes`。

**独立测试**：执行 dry-run 统计候选行，确认目标表和字段范围正确；执行真实清空后运行 verify，确认候选量为 0。

**验收场景**：

1. **Given** 生产库目标字段组的 `*_md5` 非空，**When** 执行 `--execute`，**Then** 仅同一字段组的 `*_mask / *_md5 / *_aes` 字段被置为 `NULL`。
2. **Given** 生产写入任务已暂停，**When** 清空脚本完成并执行 verify，**Then** 45 组目标字段的非空候选量均为 0。

### 用户故事 2 - 小批量低锁风险执行（优先级：P1）

运维人员需要避免在线大表被长时间锁住，脚本必须按主键小批量提交。

**独立测试**：检查执行器 SQL 逻辑，确认每批先按 `id` 取最多 500 条，再使用 `id IN (...)` 更新并立即提交。

**验收场景**：

1. **Given** 任一目标表有超过 500 行候选数据，**When** 执行真实清空，**Then** 脚本分多批提交，每批最多更新 500 行。
2. **Given** 出现锁等待或死锁，**When** 单批更新失败，**Then** 脚本最多重试 3 次，仍失败则中止并写出可恢复报告。

## 历史问题防漏分析

- 关键参数来源和赋值时机：
  - 目标表和字段：从 `069` 最终 DDL 解析生成；执行前通过 manifest 固定并校验数量。
  - profile：默认 `prod-mysql`；执行器启动时检查 `environment=prod`、`allow_write=true`。
  - 批大小：默认 500；执行时写入报告。
- 下游读取字段清单：
  - 执行器只读取 `information_schema`、目标表主键 `id` 和按 `*_md5` 统计的候选数量；不读取手机号安全字段真实值。
  - 执行器只写 `mask/md5/aes` 目标列为 `NULL`。
- 空对象 / 占位对象风险：
  - 不涉及业务 DTO、MQ、Redis、Feign 或 FC 请求。
- 调用顺序风险：
  - 必须先 `--write-artifacts`/静态校验，再 `database-sql-skill analyze`，再 dry-run，再确认备份和暂停写入，最后 execute 和 verify。
- 旧逻辑保持：
  - 不改 069 DDL，不改回填接口，不改源手机号列，不改业务查询逻辑。
- 需要用户确认的设计选择：
  - 已确认清空全部 45 组等价字段，执行前暂停写入任务，批大小默认 500。

## 边界情况

- 目标字段组的 `*_md5` 为空或空字符串：跳过，不更新，即使同组 `mask/aes` 有值也不作为本次候选。
- 单表无候选行：记录 `candidate_rows=0`，继续后续表。
- 锁等待或死锁：单批回滚并重试；超过重试次数后中止，保留已提交批次并输出失败报告。
- 执行中断：再次执行会从剩余非空候选行继续处理，已清空行不会重复更新。
- 有写入任务未暂停：脚本可以执行，但 verify 可能出现残留；本规格要求生产 execute 前暂停写入任务。

## 需求

### 功能需求

- **FR-001**：系统 MUST 从 `069` 最终 DDL 生成并校验 44 张表、45 组字段、135 个目标列。
- **FR-002**：系统 MUST 清空 `phone_*`、`app_phone_*`、`receiver_phone_*`、`reciver_phone_*` 的 `mask/md5/aes` 字段组。
- **FR-002a**：系统 MUST 只按同一字段组的 `*_md5 IS NOT NULL AND *_md5 <> ''` 选择候选行。
- **FR-003**：系统 MUST 默认 dry-run，真实写入必须显式传 `--execute` 和合规 `--confirm`。
- **FR-004**：系统 MUST 使用主键 `id` 小批量更新，默认每批 500 行，每批提交一次。
- **FR-005**：系统 MUST 设置 `innodb_lock_wait_timeout=5`，并对锁等待/死锁做有限重试。
- **FR-006**：系统 MUST NOT 修改源手机号字段、表结构、索引、业务代码或回填接口。
- **FR-007**：执行报告 MUST 不包含手机号、MD5、AES 或掩码字段真实值。

## 成功标准

- **SC-001**：manifest 静态校验结果为 44 张表、45 组字段、135 个目标列。
- **SC-002**：`database-sql-skill analyze` 对 DML 模板识别为 `dml`。
- **SC-003**：dry-run 报告记录生产候选量，且不输出敏感字段值。
- **SC-004**：execute 后 verify 报告显示全部目标字段组非空候选量为 0。

## 假设

- 生产 RDS 备份/PITR 由执行人在线上窗口前确认，本规格不导出敏感字段值作为回滚来源。
- 会重新写入这些字段的回填任务或业务任务可在执行窗口内暂停。
- 目标表主键均为 `id`；已通过生产只读元数据检查确认。
