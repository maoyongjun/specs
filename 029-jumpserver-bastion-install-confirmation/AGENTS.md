# 029-jumpserver-bastion-install-confirmation

本目录保存 Jumpserver 堡垒机安装确认与使用指引的 Spec Kit 文档。该任务属于纯运维确认，不涉及业务代码改动。

## 作用范围

- 规格目录：`C:\workspace\ju-chat\specs\029-jumpserver-bastion-install-confirmation`
- 目标对象：Jumpserver 堡垒机服务、部署主机、容器、端口、登录入口和权限配置
- 相关环境：生产区、测试区、内网管理区或其他实际部署区

## 当前目标

- 确认目标环境中是否已安装 Jumpserver，以及对应的部署区/实例归属。
- 确认服务是否运行正常，访问入口、端口和登录方式是否可用。
- 给出 Jumpserver 的标准使用指引，便于后续运维和业务用户按步骤接入。

## 执行原则

- 先确认事实，再写结论；不得只凭口头信息判断“已安装”。
- 安装确认必须覆盖至少一种可验证证据：systemd 服务、Docker 容器、K8s Pod、监听端口、健康检查页面或登录入口。
- 如果发现多个 Jumpserver 实例，必须先区分部署区/环境，再分别记录，不得混为一个结论。
- 不得在规格文档里写入真实账号、密码、私钥、Token、二维码或其他敏感凭据。
- 如果实际部署方式不明确，优先按 systemd / Docker Compose / K8s 三类常见方式分别确认，再结合现场实际归并。
- 任何会涉及重启、升级、重装、改配置或改权限的动作，都不属于本次确认任务，必须单独确认后再执行。

## 强制门禁

- 关键参数必须追踪来源和确认方式。
- 不允许用“应该有”“大概是”代替安装证据。
- 不允许把访问指引和安装确认混写成无结论的说明书。
- 如果无法确认部署区、登录入口或权限边界，必须在 `spec.md` 的风险与假设中明确写出。

## 重点信息项

- 服务名称：`jumpserver` 或现场实际注册名
- 部署区：生产区 / 测试区 / 内网管理区 / 其他实际命名
- 访问入口：域名或 IP、HTTP/HTTPS 端口、登录路径
- 部署形态：systemd / Docker Compose / K8s / 其他
- 认证方式：本地账号 / LDAP / SSO / 企业微信 / MFA / 其他
- 资产范围：主机、资产组、账号模板、网域、权限策略
