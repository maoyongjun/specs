# 功能规格：Jumpserver 堡垒机安装确认与使用指引

**功能目录**：`029-jumpserver-bastion-install-confirmation`  
**创建日期**：`2026-05-25`  
**状态**：Draft  
**输入**：用户要求在 `C:\workspace\ju-chat\specs` 创建 Spec Kit 文档，完成纯运维任务：确认目标环境是否已安装 Jumpserver 堡垒机服务，确认服务所在的部署区/实例，并提供该服务的使用指引。

## 1. 背景

运维场景中经常需要先确认堡垒机是否已经部署，再决定后续的接入、账号开通、资产授权或故障定位步骤。当前需求不是开发新功能，而是建立一份可执行、可复核的安装确认与使用说明，避免“以为已安装、实际上未部署”或“知道有堡垒机、但没人会用”的情况。

本次文档关注的是 Jumpserver 堡垒机本身：

- 它是否已经安装并可用
- 它运行在哪个部署区/环境
- 它的访问入口、认证方式和使用流程
- 如果未安装、运行异常或存在多实例，应该如何记录与分流

## 2. 目标

1. 明确目标环境中是否存在 Jumpserver 堡垒机服务。
2. 明确该服务属于哪个部署区/环境或哪一台/哪一组实例。
3. 明确服务是否处于可访问、可登录、可发起会话的状态。
4. 给出可执行的 Jumpserver 使用指引，覆盖登录、找资产、发起连接、结束会话和常见失败处理。
5. 将确认结果沉淀为标准化运维文档，便于后续交接和复核。

## 3. 范围

### 3.1 包含内容

- Jumpserver 服务是否已安装的确认
- Jumpserver 部署区/实例归属确认
- 访问入口确认，包括域名、IP、端口和路径
- 部署形态确认，包括 systemd、Docker Compose、K8s 或其他现场方式
- 登录与使用指引
- 基础故障排查路径

### 3.2 不包含内容

- 不在本次任务中执行安装、升级、重装或版本迁移
- 不在本次任务中修改配置、重启服务或变更权限
- 不在本次任务中发放真实账号密码、私钥或 Token
- 不在本次任务中编写自动化巡检脚本或改造 Jumpserver 本身

## 4. 用户场景与测试 *(必填)*

### 用户故事 1 - 确认是否已安装 Jumpserver（优先级：P1）

运维人员需要快速确认目标环境里是否真的已经安装了 Jumpserver，并知道它属于哪个部署区，避免在错误环境里排查。

**独立测试**：在目标主机或目标环境上执行服务、容器、端口和入口检查，能够得到明确的“已安装 / 未安装 / 无法确认”结论，并给出证据来源。

**验收场景**：

1. **Given** 目标环境存在 Jumpserver 服务，**When** 执行安装确认，**Then** 文档能写出服务形态、部署区、访问入口和证据来源。
2. **Given** 目标环境存在多个候选实例，**When** 执行安装确认，**Then** 文档能区分每个实例的环境归属，不把不同区混为一谈。

### 用户故事 2 - 学会使用 Jumpserver（优先级：P1）

使用者需要知道如何登录 Jumpserver、如何找到资产、如何发起连接、如何退出会话，以及遇到权限或网络问题时先看哪里。

**独立测试**：按照文档中的步骤，能够完成一次完整的登录到退出流程，并识别关键页面和关键操作入口。

**验收场景**：

1. **Given** 用户有合法账号与授权，**When** 按照文档操作，**Then** 能登录并看到可用资产或资产组。
2. **Given** 用户已登录并具备访问权限，**When** 按文档发起会话，**Then** 能进入目标主机或目标系统，并能正常退出会话。

### 用户故事 3 - 未安装或状态异常时可继续推进（优先级：P2）

运维人员在确认失败时，需要知道如何记录问题、如何区分“未安装”和“已安装但不可用”，以及下一步应该找谁确认。

**独立测试**：在服务不存在、端口不通或登录失败的情况下，文档仍能指导完成问题归档和后续确认。

**验收场景**：

1. **Given** 未发现 Jumpserver 服务，**When** 按文档检查，**Then** 能明确输出“未安装”或“未确认”，并列出下一步动作。

## 5. 历史问题防漏分析 *(强制)*

- 关键参数来源和赋值时机：
  - `部署区/环境名`：来源于现场服务器命名、CMDB、部署清单或运维交接信息；赋值时机在确认入口前。
  - `服务名/容器名`：来源于 systemd、Docker、K8s 或配置文件；赋值时机在服务发现阶段。
  - `访问入口`：来源于域名、IP、反向代理或登录页面；赋值时机在服务可达性确认阶段。
  - `认证方式`：来源于登录页、运维交接或安全策略；赋值时机在登录指引阶段。

- 下游读取字段清单：
  - 安装确认结论读取 `服务状态`、`部署区`、`访问入口`、`证据来源`。
  - 使用指引读取 `登录方式`、`资产入口`、`会话入口`、`退出方式`、`常见异常处理`。

- 空对象 / 占位对象风险：
  - 是否存在 `new XxxDto()`、空 JSON、空 Map 或只写“已安装”但没有证据：存在风险，必须避免。

- 调用顺序风险：
  - 是否存在先下结论后补证据、先写使用指引后确认入口：存在风险，必须先确认事实再定稿。

- 旧逻辑保持：
  - 已有的部署区命名、访问入口、权限边界、审计保留和账号体系必须保持原口径，不因文档整理而擅自改写。

- 需要用户确认的设计选择：
  - 如果现场存在多个 Jumpserver 实例，需要用户或运维负责人确认“本次要确认哪一个区/哪一套实例”。
  - 如果当前环境无法通过任一证据链确认安装状态，需要确认是否扩大检查范围到网络、负载均衡或上游反向代理。

## 6. 边界情况

- 只看到进程但没有服务注册信息，暂时记为“疑似已安装，需进一步确认”。
- 服务已安装但页面无法访问，需区分端口、防火墙、反向代理和应用健康问题。
- 页面可访问但登录失败，需区分账号权限、认证源、MFA 和 SSO 问题。
- 现场有测试区和生产区两个 Jumpserver，必须分开记录，不能用同一个入口描述两个环境。
- 如果 Jumpserver 部署在容器或 K8s 中，必须明确命名空间、容器名或服务暴露方式。

## 7. 需求 *(必填)*

### 功能需求

- **FR-001**：系统 MUST 能确认目标环境是否安装了 Jumpserver，并输出明确结论：已安装、未安装或暂无法确认。
- **FR-002**：系统 MUST 记录 Jumpserver 的部署区/实例归属、访问入口和部署形态。
- **FR-003**：系统 MUST 提供 Jumpserver 的标准使用指引，包括登录、资产选择、发起会话、退出会话和审计查看。
- **FR-004**：系统 MUST 明确常见异常的排查方向，包括服务未启动、端口不通、认证失败、权限不足和多实例混淆。
- **FR-005**：系统 MUST NOT 在确认任务中修改配置、重启服务、变更权限或泄露敏感凭据。
- **FR-006**：文档 MUST 记录证据来源，不允许只有主观判断没有验证依据。

## 8. 成功标准 *(必填)*

- **SC-001**：阅读文档后，能够在目标环境中判断 Jumpserver 是已安装、未安装还是暂无法确认。
- **SC-002**：阅读文档后，能够明确 Jumpserver 属于哪个部署区/实例，并知道对应的访问入口。
- **SC-003**：阅读文档后，使用者能够完成一次标准的 Jumpserver 登录与会话访问流程。
- **SC-004**：阅读文档后，能够根据常见异常提示快速区分服务问题、网络问题和权限问题。
- **SC-005**：文档不包含真实密码、私钥、Token 等敏感信息。

## 9. 假设

- 目标环境可能采用 systemd、Docker Compose、K8s 或其他方式部署 Jumpserver，实际确认时以现场部署方式为准。
- 登录方式可能是本地账号、企业统一身份认证、MFA 或其他安全策略，文档只描述流程，不固定具体认证产品。
- 部署区命名可能是生产区、测试区、内网管理区或其他自定义名称，文档中以现场实际命名为准。
- 若现场无法确认唯一实例，默认把“多实例并存”视为风险而不是误判为单实例。
